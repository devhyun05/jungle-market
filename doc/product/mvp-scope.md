# MVP 범위 정의서

## 1. 문서 목적

이 문서는 크래프톤 정글 내부 중고거래 웹서비스의 MVP 범위를 명확히 정의하기 위해 작성한다.

본 문서는 다음 의사결정의 기준으로 사용한다.

- MVP에 포함할 기능과 제외할 기능 결정
- 개발 우선순위 판단
- GitHub Projects 태스크 분리
- 화면 설계 범위 확정
- ERD 및 API 명세 작성 범위 확정
- 일정 관리 및 기능 변경 판단

## 2. MVP 목표

MVP의 목표는 크래프톤 정글 내부 구성원이 중고 물품을 쉽고 안전하게 등록, 탐색, 문의, 거래 상태 관리할 수 있는 최소 기능의 웹서비스를 제공하는 것이다.

MVP에서는 외부 중고거래 서비스처럼 결제, 배송, 추천, 복잡한 평판 시스템을 제공하지 않는다.

대신 정글 내부 사용자에게 필요한 다음 문제를 우선 해결한다.

- 상품 정보가 여러 채널에 흩어지는 문제
- 원하는 상품을 찾기 어려운 문제
- 판매 상태를 확인하기 어려운 문제
- 판매자와 구매자 간 문의가 흩어지는 문제
- 부적절한 게시글과 신고를 관리하기 어려운 문제

## 3. MVP 대상 사용자

| 사용자 유형 | 설명                                               |
| ----------- | -------------------------------------------------- |
| 일반 사용자 | 상품을 등록하거나 상품을 탐색하는 정글 내부 구성원 |
| 판매자      | 상품을 등록하고 판매 상태를 관리하는 사용자        |
| 구매자      | 상품을 검색하고 판매자에게 문의하는 사용자         |
| 관리자      | 신고와 게시글을 관리하는 운영 담당자               |

## 4. MVP 핵심 가치

MVP는 다음 가치를 제공해야 한다.

1. 내부 구성원만 사용할 수 있는 거래 공간을 제공한다.
2. 사용자가 상품을 쉽게 등록하고 찾을 수 있게 한다.
3. 상품의 판매 상태를 명확히 확인할 수 있게 한다.
4. 구매자가 판매자에게 문의할 수 있게 한다.
5. 부적절한 게시글을 신고하고 관리자가 조치할 수 있게 한다.

## 6. MVP 제외 범위

MVP에서는 다음 기능을 제외한다.

| 제외 기능          | 제외 이유                                                 |
| ------------------ | --------------------------------------------------------- |
| 결제               | 내부 직거래 중심 서비스이므로 초기 범위에서 제외한다.     |
| 에스크로           | 결제 기능이 없으므로 MVP에서는 제외한다.                  |
| 택배 거래          | 정글 내부 또는 근거리 직거래를 우선한다.                  |
| 외부 사용자 거래   | 내부 커뮤니티 거래에 집중한다.                            |
| 실시간 채팅 고도화 | 초기에는 상품별 문의 기능으로 대체한다.                   |
| 이미지 첨부 메시지 | 문의 기능의 복잡도를 줄이기 위해 제외한다.                |
| 자동 추천 알고리즘 | 초기에는 검색과 필터 중심으로 탐색 문제를 해결한다.       |
| 가격 시세 분석     | 거래 데이터가 충분히 쌓인 이후 검토한다.                  |
| 복잡한 평판 점수   | MVP에서는 신고와 관리자 조치 중심으로 신뢰 문제를 다룬다. |
| AI 상품 설명 생성  | 핵심 문제 해결과 직접 관련성이 낮아 제외한다.             |
| 다국어 지원        | 초기 사용자는 정글 내부 한국어 사용자를 기준으로 한다.    |
| 모바일 앱          | 반응형 웹으로 우선 대응한다.                              |

## 7. MVP 기능 상세 범위

## 7.1 인증

### 포함

- 사용자는 정글 내부 구성원임을 확인할 수 있는 방식으로 로그인할 수 있어야 한다.
- 비로그인 사용자는 서비스 주요 기능에 접근할 수 없어야 한다.
- 로그인한 사용자만 상품 목록, 상세, 등록, 문의 기능을 사용할 수 있어야 한다.

### 제외

- 소셜 로그인 다중 제공
- 복잡한 권한 그룹 관리
- 사용자 자체 회원가입 승인 워크플로우

## 7.2 상품 등록/관리

### 포함

- 사용자는 상품을 등록할 수 있어야 한다.
- 상품 등록 시 제목, 가격, 설명, 카테고리, 상품 상태, 거래 장소를 입력할 수 있어야 한다.
- 사용자는 상품 이미지를 1개 이상 등록할 수 있어야 한다.
- 작성자는 자신이 등록한 상품을 수정할 수 있어야 한다.
- 작성자는 자신이 등록한 상품을 삭제할 수 있어야 한다.
- 작성자는 상품 상태를 판매중, 예약중, 거래완료로 변경할 수 있어야 한다.

### 제외

- 상품 끌어올리기
- 가격 변경 이력
- 임시 저장
- 대량 상품 등록

## 7.3 상품 탐색/검색

### 포함

- 사용자는 상품 목록을 볼 수 있어야 한다.
- 사용자는 상품 상세 정보를 볼 수 있어야 한다.
- 사용자는 키워드로 상품을 검색할 수 있어야 한다.
- 사용자는 카테고리별로 상품을 필터링할 수 있어야 한다.
- 사용자는 판매중인 상품만 볼 수 있어야 한다.
- 사용자는 최신순으로 상품 목록을 정렬할 수 있어야 한다.

### 제외

- 추천 상품
- 인기 상품 랭킹
- 가격 범위 필터
- 복합 정렬
- 최근 본 상품

## 7.4 문의/거래

### 포함

- 구매자는 상품 상세 화면에서 판매자에게 문의할 수 있어야 한다.
- 판매자는 자신이 등록한 상품에 대한 문의를 확인할 수 있어야 한다.
- 문의는 상품 단위로 관리되어야 한다.
- 사용자는 본인이 참여한 문의 내역을 확인할 수 있어야 한다.

### 제외

- 실시간 채팅
- 메시지 읽음 표시
- 메시지 이미지 첨부
- 음성/영상 통화
- 자동 거래 확정

## 7.5 신고

### 포함

- 사용자는 부적절한 상품 게시글을 신고할 수 있어야 한다.
- 신고 시 신고 사유를 선택하거나 입력할 수 있어야 한다.
- 관리자는 신고 목록을 확인할 수 있어야 한다.
- 관리자는 신고된 게시글을 숨김 처리할 수 있어야 한다.

### 제외

- 자동 신고 판별
- 신고 누적 자동 제재
- 분쟁 중재 프로세스
- 사용자 간 차단

## 7.6 관리자

### 포함

- 관리자는 관리자 페이지에 접근할 수 있어야 한다.
- 일반 사용자는 관리자 페이지에 접근할 수 없어야 한다.
- 관리자는 전체 상품 목록을 확인할 수 있어야 한다.
- 관리자는 신고된 게시글 목록을 확인할 수 있어야 한다.
- 관리자는 부적절한 상품을 숨김 처리할 수 있어야 한다.

### 제외

- 관리자 공지사항 기능
- 사용자 정지/정지 해제
- 관리자 상세 감사 로그 화면
- 운영 통계 대시보드

## 8. MVP 주요 화면

MVP에서 필요한 화면은 다음과 같다.

| 화면                  | 설명                                                            |
| --------------------- | --------------------------------------------------------------- |
| 로그인 화면           | 내부 사용자가 서비스에 로그인하는 화면                          |
| 상품 목록 화면        | 상품 목록, 검색, 카테고리 필터, 판매중 필터를 제공하는 화면     |
| 상품 상세 화면        | 상품 정보, 이미지, 판매자 정보, 문의, 신고 기능을 제공하는 화면 |
| 상품 등록 화면        | 상품 정보를 입력하고 이미지를 업로드하는 화면                   |
| 상품 수정 화면        | 작성자가 기존 상품 정보를 수정하는 화면                         |
| 마이페이지            | 내가 등록한 상품과 문의 내역을 확인하는 화면                    |
| 문의 화면             | 상품별 문의 메시지를 확인하고 작성하는 화면                     |
| 관리자 상품 관리 화면 | 전체 상품을 확인하고 숨김 처리하는 화면                         |
| 관리자 신고 관리 화면 | 신고 목록을 확인하고 처리하는 화면                              |

| 데이터         | 제외 이유                                             |

| AdminActionLog | 초기에는 최소한의 관리자 기능을 우선 구현한다.        |
| Review         | 거래 후기와 평판 시스템은 MVP 이후 검토한다.          |

## 11. MVP 성공 기준

MVP는 다음 조건을 만족하면 성공으로 판단한다.

- 내부 사용자가 로그인할 수 있다.
- 로그인한 사용자가 상품을 등록할 수 있다.
- 사용자가 상품 목록과 상세 정보를 조회할 수 있다.
- 사용자가 검색 또는 필터를 통해 원하는 상품을 찾을 수 있다.
- 상품의 판매 상태가 판매중, 예약중, 거래완료로 표시된다.
- 구매자가 상품 상세 화면에서 판매자에게 문의할 수 있다.
- 판매자가 상품에 대한 문의를 확인할 수 있다.
- 사용자가 부적절한 게시글을 신고할 수 있다.
- 관리자가 신고 목록을 확인할 수 있다.
- 관리자가 부적절한 게시글을 숨김 처리할 수 있다.

| 인증 방식 미확정             | 내부 사용자 확인 방식에 따라 개발 범위가 달라질 수 있다.    | 개발 전 인증 방식을 먼저 결정한다.                     |
| 문의 방식 범위 증가          | 실시간 채팅 요구가 생기면 개발 복잡도가 증가한다.           | MVP에서는 상품별 문의 방식으로 제한한다.               |
| 이미지 업로드 비용 및 저장소 | 이미지 저장 방식에 따라 인프라 비용과 구현 방식이 달라진다. | 초기에는 단순한 이미지 저장 구조를 사용한다.           |
| 신고 처리 기준 불명확        | 운영 기준이 없으면 관리자 판단이 어려워진다.                | 운영 정책 문서를 별도로 작성한다.                      |
| 기능 추가 요청 증가          | 프로젝트 중간에 범위가 커질 수 있다.                        | 본 문서의 제외 범위를 기준으로 후속 버전으로 분리한다. |

- 가격 낮은순/높은순 정렬
- 실시간 채팅

- 거래 후기
- 간단한 매너 점수

- 운영 통계 대시보드
